[^off211]: Closed by making the rate-limit `Map<ip, {count,resetAt}>` a true LRU. Pre-fix the eviction policy was insertion-order-only — `rateLimit` mutated `entry.count` in place and never re-inserted, so the bucket created at time T sat at the FRONT of `Map`'s iteration order regardless of how active it was. When the cap was hit (a scanner rotating cold IPs), `pruneRateLimitMap`'s `keys().next().value` cap-evict loop targeted the HOTTEST client first, freed its quota, and let it renew. The S-offchain-02 header comment's "safe because each entry already expires after RATE_LIMIT_WINDOW_MS" reasoning didn't hold for the cap-hit branch — that branch fires PRECISELY when TTL pruning hasn't caught up yet. Fix: every touch in `rateLimit` (fresh-window create, count-bump on a non-rejected hit, AND the rate-limited 429 rejection) does `delete(ip)` then `set(ip, entry)`, moving the entry to the END of `Map`'s insertion-ordered iteration. The pruner still pops from the FRONT — which is now the LEAST-recently-used end. Hand-rolled rather than introducing an `lru-cache` workspace dep: the data shape was already a `Map`, the eviction trigger was already insertion-order, and the only missing piece was recency-bump-on-touch (5 lines of behavior change inside an existing audit-comment block). Regression tests: `src/x402-relay/test/off-211-216.test.ts` covers (1) fresh IPs land at the END of iteration order, (2) touching an existing IP moves it to the END (not in-place update), (3) count-bump path bumps recency, (4) the 429 rejection path also bumps recency (a hot rejected client must not be evicted ahead of a cold one-shot), (5) the eviction-order observable — `keys().next().value` is the truly LRU IP, not the hottest one — and the negative assertion that pre-fix-shape ordering does NOT hold, (6) TTL eviction still works on unexpired entries (LRU change must not regress the TTL pruner).

[^off210]: Closed by reordering the `failures = 0` reset in `startConnectionHeartbeat` (`src/indexer/index.ts`) so the counter is cleared ONLY after `opts.onConnectionLost(reason)` returns successfully. Pre-fix the reset happened BEFORE the call: a callback that itself threw (e.g. a `removeOnLogsListener` reject on a flaky network, or a synchronous re-subscribe error) had its own failure invisibly swallowed AND the heartbeat counter was already at zero — so the next outage tick had to climb the threshold from scratch instead of firing immediately. [^off212]: Closed by introducing a Postgres session-scoped advisory lock primitive (`acquireIndexerWriterLock` in `src/indexer/postgres-store.ts`) and wiring it into the indexer's `main()` boot path in `src/indexer/index.ts`. **Lock primitive**: at boot, the indexer takes `pg_try_advisory_lock(INDEXER_WRITER_LOCK_KEY)` on a borrowed pool client. The lock key is a fixed bigint (`0x4145_5046_1212_0212`) — stable across releases so a new indexer process always conflicts with an old one running against the same DB. The lock is **session-scoped** (not transaction-scoped) so it lasts as long as the holding client holds the connection — which is the indexer's process lifetime. **What holds it**: the borrowed `PoolClient` returned by `acquireIndexerWriterLock` — it is NOT released back to the pool after the `pg_try_advisory_lock` query (doing so would release the lock immediately because session-scoped advisory locks bind to the session that took them). **What releases it on crash**: Postgres itself. When the indexer process exits — graceful SIGINT, SIGTERM, OOM-kill, segfault, host reboot — the connection closes and PG releases the lock automatically. No external cleanup required, no stale-lock recovery, no TTL tuning. **Graceful-shutdown release**: the SIGINT handler calls `writerLock.release()` which issues `pg_advisory_unlock` and returns the client to the pool — an optimisation for clean exit, idempotent and safe to call when the lock was never acquired. **Boot policy**: when PG is configured AND the lock is denied (another indexer holds it), `main()` logs FATAL and `process.exit(1)`s. The `INDEXER_ALLOW_NO_WRITER_LOCK=1` env opt-out exists so a one-off debug run (read-only inspection, schema-parity check) can proceed without owning the lock; the default is fail-closed. **SQLite-only fallback**: when `INDEXER_PG_URL` is unset, no shared backend exists to coordinate on, but the SQLite WAL itself rejects a concurrent writer with `SQLITE_BUSY` — so the lock helper returns `acquired=false` with a no-op release and the boot path proceeds. The audit's punchlist suggested `proper-lockfile` for SQLite mode; that dependency is not in the workspace and adding it for a corner case the SQLite WAL already covers would be a net regression. Regression test: `src/indexer/test/off-208-215-bundle.test.ts` "OFF-212" describe-block covers (1) `DisabledPostgresStore` returns `acquired=false` with a no-op release, (2) `INDEXER_WRITER_LOCK_KEY` is a stable non-zero bigint, (3) two stores sharing the same lock state — first acquires, second is denied; after first releases, a third acquires (the load-bearing pg_try_advisory_lock semantic, asserted via a hand-rolled in-memory mock since pg-mem 3.x does not implement advisory locks), and (4) `release()` is idempotent. Behavioural enforcement against a real PG server (the actual `pg_try_advisory_lock` cluster-state) is verifiable via an operator-driven run with `INDEXER_PG_TEST_URL` set against a live Postgres — same opt-in posture as OFF-217.
