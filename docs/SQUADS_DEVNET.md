# Squads v4 Multisig — Devnet Bootstrap (v1)

Tracking reference for the Agenomics Protocol devnet Squads v4 multisig.
This multisig is a **v1 dry-run** intended to serve as the `AEP_PROTOCOL`
SAS credential authority (per [ADR-063][adr]) on devnet.

> **Scope (v1 devnet only)**
>
> - Cluster: `devnet`
> - Squads v4 SDK: `@sqds/multisig ^2.1.0`
> - Threshold: **2-of-3**
> - Role: placeholder authority for the `AEP_PROTOCOL` SAS credential when
>   the ADR-063 §5 devnet ceremony runs.
>
> **Explicitly NOT in scope for this v1 bootstrap:**
>
> - Transferring program upgrade authority (Registry / Settlement / Vault)
>   to the multisig.
> - Any mainnet action.
> - Creating SAS credentials (that is the follow-up ceremony — ADR-063 §5).
> - The ADR-063 mainnet composition (3-of-5 with role slots). Devnet v1
>   uses a throwaway 2-of-3 for mechanical rehearsal.

## Running the bootstrap

The script is idempotent: if a live multisig is already recorded in
`scripts/.squads-devnet.json`, it exits without doing anything.

```bash
# Prereqs:
# - Solana CLI configured to devnet (`solana config set --url devnet`)
# - Funder wallet at ~/.config/solana/id.json with >= 0.2 SOL on devnet
npm install
npx tsx scripts/bootstrap-squads-devnet.ts
```

On first run the script will:

1. Load signer 1 from `~/.config/solana/id.json`.
2. Generate signers 2 and 3 at `.keys/squads-signer-2.json` and
   `.keys/squads-signer-3.json` if they are not already there.
3. Top up signers 2 and 3 to ~0.02 SOL each from signer 1.
4. Derive a fresh `createKey`, then the multisig PDA from it.
5. Read the Squads `ProgramConfig` PDA to resolve the `treasury` pubkey
   required by `multisigCreateV2`.
6. Submit the `multisigCreateV2` transaction (signer 1 pays + creates).
7. Read back the on-chain `Multisig` account and assert
   `threshold == 2` and `members.length == 3`.
8. Write `scripts/.squads-devnet.json` with all public fields and the
   transaction signature.

The config-shape is guarded by
`tests/squads-bootstrap-config.test.ts` (skipped until the file exists).

## Proposing a transaction via the multisig

Once bootstrapped, any transaction that requires multisig approval
follows the Squads v4 proposal / vote / execute flow. See the official
[Squads v4 SDK docs][sdk] for full examples. The 2-of-3 shape on devnet
means the minimum path is:

1. **Signer 1 creates a vault transaction and the associated proposal.**
   ```ts
   await multisig.rpc.vaultTransactionCreate({
     connection,
     feePayer: signer1,
     multisigPda,
     transactionIndex: nextIndex,
     creator: signer1.publicKey,
     vaultIndex: 0,
     ephemeralSigners: 0,
     transactionMessage, // the tx the multisig should eventually execute
     addressLookupTableAccounts: [],
   });
   await multisig.rpc.proposalCreate({
     connection,
     feePayer: signer1,
     multisigPda,
     transactionIndex: nextIndex,
     creator: signer1,
   });
   ```
2. **Signer 1 votes approve** (via `proposalApprove`). The proposal is
   now 1/2 toward threshold.
3. **Either signer 2 or signer 3 votes approve** via `proposalApprove`.
   Threshold met, proposal becomes `Approved`.
4. **Signer 1 (or any member) executes** via `vaultTransactionExecute`.
   The inner transaction lands on-chain, signed by the multisig vault
   PDA.

A reject instead of approve burns an approval slot; after 2 rejects the
proposal is cancelled. Cancel and re-propose if the content was wrong.

## Rotation

Devnet v1 does not have a formal rotation policy — the signer keypairs
are throwaway and can be re-generated by deleting `.keys/` and
re-running the bootstrap (which will also replace the multisig, since
the old `multisigPda` will not match a newly-derived `createKey`).

Mainnet / production rotation is governed by [ADR-063][adr]:

- **Routine rotation:** §4 — 14 day notice window, supermajority (4-of-5
  on the mainnet 3-of-5 multisig), atomic add-then-remove, post-rotation
  attestation audit.
- **Emergency rotation (compromise):** §6.1 — 0 day notice, simple
  majority + auditor co-sign, mandatory retroactive audit of
  attestations signed in the suspected-compromise window.
- **Authority compromise (threshold-loss):** §6.2 — escalates to a new
  multisig created by the program-upgrade multisig; old credential
  permanently suspended; historical attestations remain valid.

For devnet, if a signer key is lost, rebuild the multisig (cheap).
