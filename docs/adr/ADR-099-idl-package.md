# ADR-099: `@agenomics/idl` — published npm package for on-chain IDL JSON and TypeScript types

## Status

Accepted

## Date

2026-04-23

**Audit-items:** ARCHITECTURE-AUDIT-2026-04-23 → items 23 & 31

## Context

The AEP protocol exposes three Anchor programs — `agent_registry`, `agent_vault`, and `settlement` — each with a corresponding IDL JSON file under `idl/`. These files are the canonical machine-readable description of every on-chain instruction, account, and error.

Before this ADR, TypeScript consumers of the IDL (mcp-server, indexer, x402-relay) had no shared source of truth:

- `mcp-server` loaded the IDL at runtime via `require()` from `target/idl/*.json`, which is a build artifact that only exists after `anchor build` runs.
- `src/indexer` and `src/x402-relay` either inlined duplicate program addresses as bare strings or used `as any` casts when importing IDL JSON.
- The `sdk/idl` workspace package (`sdk/idl/`) already existed and exported cluster-keyed program IDs (`PROGRAM_IDS`, `getProgramIds`), but did not export the IDL JSON itself.

The result was that the IDL — the single most important protocol artefact — was consumed in at least three different ways across the codebase, with no lint or type enforcement that consumers were using the correct version.

ADR-088 (typed Anchor program clients) established that `Program<IDL>` should be parameterised with the concrete IDL type. ADR-088 noted that factoring the IDL into a publishable package was the correct long-term solution and designated it as ADR-099.

## Decision

Extend the existing `sdk/idl` workspace package (`@agenomics/idl`) to export the IDL JSON files as TypeScript consts alongside the program ID map already shipped by the package.

Concretely:

1. Copy `idl/agent_registry.json`, `idl/agent_vault.json`, and `idl/settlement.json` into `sdk/idl/src/idl/` so the package is self-contained and does not depend on `target/idl/` build artefacts.

2. Add thin TypeScript wrapper modules (`src/idl/agent_registry.ts`, `src/idl/agent_vault.ts`, `src/idl/settlement.ts`) that import the JSON using `with { type: "json" }` (ES2022 import attributes) and re-export a named const (`AgentRegistryIdl`, `AgentVaultIdl`, `SettlementIdl`).

3. Re-export all three consts from `src/index.ts` alongside the existing `PROGRAM_IDS` / `getProgramIds` / `Cluster` exports so consumers have a single import target.

4. Enable `resolveJsonModule: true` in `sdk/idl/tsconfig.json` so TypeScript can type the JSON shape.

5. Add a `typecheck` script (`tsc --noEmit`) to the package's `scripts`.

Consumers cast to `Idl` from `@coral-xyz/anchor` at the call site:

```ts
import type { Idl } from "@coral-xyz/anchor";
import { AgentRegistryIdl } from "@agenomics/idl";

const program = new Program(AgentRegistryIdl as unknown as Idl, provider);
```

The `@agenomics/idl` package itself does not take a dependency on `@coral-xyz/anchor`, keeping it lightweight for consumers (indexer, relayer) that don't construct `Program` instances.

## Alternatives

**Alternative A — import IDL from `target/idl/*.json` in each consumer.** This is the status quo. The problem is that `target/` is a build artefact directory: it is absent on fresh checkouts until `anchor build` runs, and its lifecycle is owned by Anchor, not npm. It cannot be published or referenced as a stable workspace package.

**Alternative B — generate TypeScript consts from the IDL via a codegen script.** Adds tooling complexity and a codegen step that must be re-run whenever the IDL changes. The Anchor IDL JSON is already a stable machine-readable format; wrapping it in a `const` is sufficient.

**Alternative C — create a new `packages/idl` package.** This was the initial approach in this implementation but was abandoned because `sdk/idl` already existed with the same package name (`@agenomics/idl`) and already exported program IDs. Creating a duplicate workspace package with the same name is a workspace conflict. Extending the existing package is the correct incremental approach.

**Alternative D — publish `@coral-xyz/anchor` IDL types as part of `@agenomics/idl`.** The typed surfaces (`AgentRegistry`, `AgentVault`, `Settlement` from `target/types/*.ts`) are generated by `anchor build` and reflect the Rust program at build time. Including them would bind the npm package to a build artefact. ADR-088 already handles this concern for `mcp-server` via `target/types/*.ts` imports; this ADR focuses on the distributable IDL JSON.

## Consequences

### Positive

- **Single source of truth for IDL JSON.** All consumers import from `@agenomics/idl`. IDL drift across the monorepo is eliminated.
- **No `anchor build` prerequisite for off-chain consumers.** The IDL JSON is vendored into `sdk/idl/src/idl/`, so the package resolves without requiring `target/idl/` to exist.
- **Typed const.** `resolveJsonModule` gives TypeScript the full shape of the IDL JSON. Consumers that need a specific field (e.g. `AgentRegistryIdl.address`) get autocompletion and type safety.
- **Package remains lightweight.** No runtime dependency on `@coral-xyz/anchor`; consumers that need `Idl` typing cast at their call site.
- **`typecheck` script.** `npm run typecheck` in `sdk/idl` verifies the package compiles cleanly in CI without a full build.

### Negative / cost

- **IDL JSON is now duplicated.** `idl/*.json` (canonical source, committed in the protocol repo) and `sdk/idl/src/idl/*.json` (package copy) must be kept in sync. A future CI check or a `scripts/` script should enforce this (e.g. `diff idl/agent_registry.json sdk/idl/src/idl/agent_registry.json`). This is acceptable because the canonical JSON is the Anchor build output, and any Rust program change that updates `idl/*.json` must also be propagated to `sdk/idl/src/idl/`.
- **Import attributes (`with { type: "json" }`) require Node ≥ 20.** All existing package scripts in the monorepo already target Node 20+; this is not a new constraint.

### Out of scope (follow-up)

- **CI diff check** to detect when `idl/*.json` and `sdk/idl/src/idl/*.json` diverge.
- **Typed `Program` exports** (`Program<AgentRegistry>` etc.) once ADR-099 consumers are confirmed and the `target/types/*.ts` lifecycle question is resolved.
- **Publishing to npm.** The package is currently `UNLICENSED` and workspace-local. Publishing to a public or private registry is a separate decision.
