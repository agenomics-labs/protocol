# Settlement devnet upgrade-authority recovery

Operational runbook for locating, transferring, and verifying the
Settlement program's BPF upgrade-authority keypair when it is not
present on the host running ceremonies.

## When to use this runbook

The Settlement program on devnet
(`9TRVbw2dvER1zDQcxwA8Puub4fLnPGstc1GGDDLTUF95`) is deployed but its
singleton `ProtocolConfig` PDA
(`EEz7JfgdJ1nVYafjpzvtXtvnyC4yHF1mayAT5EUxz3oh`) was never
initialized. Until `initialize_protocol_config` runs, every
escrow-shaped instruction (`create_escrow`, `accept_task`,
`submit_milestone`, `approve_milestone`, `resolve_dispute`) reverts
with Anchor 3012 (`AccountNotInitialized`) for `protocol_config`.

On-chain constraint (`programs/settlement/src/contexts.rs:595-622`):

```text
constraint = program_data.upgrade_authority_address == Some(payer.key())
```

Only the Settlement program's upgrade authority can run the one-shot
init. On-chain readback:

```bash
solana program show 9TRVbw2dvER1zDQcxwA8Puub4fLnPGstc1GGDDLTUF95 -u devnet
# Authority: 8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz
```

That keypair is not present in `.keys/` or `~/.config/solana/` on the
primary dev host. This runbook covers finding it on another machine
the operator owns.

For background on why the on-chain constraint is shaped this way, see
`docs/PROTOCOL_AUTHORITY_OPERATIONS.md` §1–§4 and
`docs/audits/DESIGN-DECISIONS-2026-04-25.md` (AUD-005).

## Step 1 — quick scan on the other machine

SSH into the other dev machine and run a narrow scan of standard
Solana keypair locations. This catches the typical case where the
keypair was generated with `solana-keygen new -o <path>` into one of
the conventional directories.

```bash
TARGET=8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz

shopt -s globstar nullglob 2>/dev/null || true
for f in \
    ~/.config/solana/*.json \
    ~/.config/solana/**/*.json \
    ~/.keys/*.json \
    ~/keypairs/*.json \
    ~/dev/**/.keys/*.json \
    ~/dev/**/keypairs/*.json \
    ~/Documents/**/*.json
do
  [ -f "$f" ] || continue
  pub=$(solana-keygen pubkey "$f" 2>/dev/null) || continue
  if [ "$pub" = "$TARGET" ]; then
    echo "FOUND: $f"
  fi
done
```

If `FOUND:` is printed, skip to step 3.

## Step 2 — wider sweep

If step 1 returns nothing, scan the full home directory. This is
slower (~30 s to several minutes depending on home size) but
exhaustive. The size filter `-size -2k` excludes large JSON files
that cannot be 64-byte keypairs.

```bash
TARGET=8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz

find ~ -type f -name "*.json" -size -2k \
  -not -path "*/node_modules/*" \
  -not -path "*/.git/*" \
  -not -path "*/target/*" \
  -not -path "*/.cache/*" \
  2>/dev/null | while read -r f; do
    pub=$(solana-keygen pubkey "$f" 2>/dev/null) || continue
    [ "$pub" = "$TARGET" ] && echo "FOUND: $f"
  done
```

If the scan completes with no `FOUND:` line, the keypair is not on
this machine. Try another machine, or fall back to step 5.

## Step 3 — transfer to the primary host

Pull the file from the primary host (do **not** push from the source
machine). Pulling makes the destination path explicit and keeps the
keypair out of any cloud-sync directory.

```bash
# Run this on the primary host (the one that will run the init):
scp <other-machine-hostname-or-ip>:<absolute-path-from-step-1-or-2> \
    /home/neo/dev/projects/protocol/.keys/settlement-upgrade-authority-devnet.json
chmod 600 /home/neo/dev/projects/protocol/.keys/settlement-upgrade-authority-devnet.json
```

Do not transit the keypair through email, Slack, Discord, or any
cloud drive that indexes contents. SCP over SSH only. The keypair is
a 64-byte JSON file with no password protection — anyone who reads it
controls the Settlement program's upgrade authority.

## Step 4 — verify

On the primary host:

```bash
solana-keygen pubkey \
  /home/neo/dev/projects/protocol/.keys/settlement-upgrade-authority-devnet.json
# expected: 8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz
```

If the printed pubkey matches `8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz`,
the recovery is complete. Proceed to "Next step" below.

If the pubkey does not match, the wrong file was copied. Re-check
step 1 / step 2 output and repeat step 3.

## Step 5 — fallbacks if not found

**Seed-phrase recovery.** If the keypair was generated by
`solana-keygen new` (default), a BIP-39 seed phrase was printed at
creation time. If that phrase was saved (password manager, paper
backup, notes app):

```bash
solana-keygen recover -o \
  /home/neo/dev/projects/protocol/.keys/settlement-upgrade-authority-devnet.json
# Enter the seed phrase + optional BIP-39 passphrase when prompted.
```

Then re-run step 4 to verify.

**Redeploy.** If the keypair is genuinely lost and no seed phrase
exists, the on-chain authority is unrecoverable — `Authority:` is
locked to the key that originally signed `solana program deploy` and
there is no recovery instruction. The only path forward is redeploy
of Settlement at a new program ID with the local operator wallet
(`BUdXA1FiWnV7ksXYodH3uEhDUhfBJ8g4UmmWdshWjTXL`) as the new upgrade
authority. This invalidates the program ID referenced in
`SUBMISSION.md`, `README.md`, and `JUDGE_RUNBOOK.md`; all three
must be updated in the same change. See
`docs/MAINNET_DEPLOY_RUNBOOK.md` §2 for the deploy choreography (the
devnet variant is the same flow without the multisig step).

## Next step — run `initialize_protocol_config`

Once the keypair is on the primary host and verified, the devnet
`initialize_protocol_config` ceremony is a one-shot transaction. The
procedure mirrors `docs/MAINNET_DEPLOY_RUNBOOK.md` §3 with two
simplifications: no Squads proposal (single-key signer), and no
threshold coordination.

Key facts to internalise before running it (from
`docs/PROTOCOL_AUTHORITY_OPERATIONS.md` §1–§4 and
`docs/audits/DESIGN-DECISIONS-2026-04-25.md`):

- **The signer of `initialize_protocol_config` becomes
  `ProtocolConfig.authority` for the lifetime of this deployment.**
  No rotation instruction exists; ADR-125 defers it to the first
  post-launch governance cycle.
- **There is no in-script warning if the wrong key signs.** The
  Anchor constraint accepts any key whose pubkey equals the upgrade
  authority. If the wrong upgrade-authority key is rotated in before
  this step, the wrong identity becomes the protocol authority
  permanently.

On devnet specifically, the consequences are reversible by redeploy
at a new program ID — so the cost of a mis-bind is "update three docs
and re-record the demo," not "lose the launch." But verify on-chain
readback before treating the demo as live:

```bash
solana account EEz7JfgdJ1nVYafjpzvtXtvnyC4yHF1mayAT5EUxz3oh -u devnet
# Bytes [8..40) of the account data must equal:
# 8vj7tBNqdZxzcMt3U2F4wQNsrvz5SJmd8FpdSLTTMqKz
```

## Storage discipline after recovery

The keypair should not live indefinitely under
`.keys/settlement-upgrade-authority-devnet.json` on a working dev
host:

- The repo's `.gitignore` excludes `.keys/`, so it will not be
  committed — but a stray `git add -f` or shell completion accident
  can leak it. Confirm `git status` shows the file as ignored, not
  untracked, before any commit.
- Long-term it belongs in a password manager or hardware-backed
  store, not on a working filesystem. Treat the local copy as a
  transient artifact for the ceremony.
- After the ceremony, the on-chain authority can be transferred to
  the operator wallet or a Squads multisig (see
  `docs/MAINNET_CHECKLIST.md` §"Transfer upgrade authority"). Devnet
  follows the same `solana program set-upgrade-authority` flow as
  mainnet, just without the multisig signing step.
